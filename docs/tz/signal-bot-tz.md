## 1. Общее описание проекта

NMNH Signal Bot — персонализированный Telegram‑бот для рассылки торговых сигналов ученикам
трейдинг‑ментора. Ментор вводит сигнал (пару и направление), бот автоматически рассчитывает
параметры сделки индивидуально под баланс каждого ученика и отправляет персональное сообщение.

> ⚠️ **ОТКРЫТОЕ РЕШЕНИЕ (способ доставки).** Исходно сообщения отправлялись от личного аккаунта
> ментора через Pyrogram userbot. Это несёт риск блокировки личного аккаунта ментора со стороны
> Telegram. Окончательный способ доставки — решение заказчика. См.
> [AUDIT.md → A‑02](../audit/AUDIT.md#a-02-userbot-pyrogram--риск-блокировки-личного-аккаунта).

Ключевые особенности системы:

- Два режима торговли: **Умеренный** (до x25) и **Турбо** (до x400).
- Баланс каждого ученика тянется автоматически через **аффилиат API WEEX** по UID аккаунта.
- Все расчёты позиции, маржи, риска и тейков — индивидуальные под каждого ученика.
- Доставка персональных сообщений (способ — см. открытое решение выше).
- Гибкий формат ввода сигнала — от минимального до полного.
- Поддержка двух языков: русский и английский.

## 2. Технический стек

| Компонент | Технология | Назначение |
|---|---|---|
| Основной язык | Python 3.10+ | Вся бизнес‑логика |
| Интерфейс ментора/ученика | aiogram 3 | Приём команд, ввод сигналов, онбординг |
| Доставка рассылки | Pyrogram userbot **или** бот (откр. решение) | Отправка сообщений ученикам |
| База данных | **PostgreSQL 15+** (см. правку) | Хранение учеников, сигналов, настроек |
| Шифрование | cryptography (AES‑256) | Защита чувствительных данных |
| Планировщик | APScheduler | Автообновление балансов |
| Хостинг | VPS (Linux/Windows), 24/7 | Постоянная работа |
| Логирование | Python logging | Логи в файл + ротация |
| WEEX API | REST API V3 (Futures) | Цены, свечи, данные рынка |
| WEEX Affiliate API | REST API V2/V3 | Баланс и данные учеников по UID |

> 🔧 **Правка БД:** исходно SQLite. Заменено на **PostgreSQL**, чтобы бот и веб‑платформа
> использовали одну СУБД и одну модель данных (см. [unified-core.md](../architecture/unified-core.md)).
> Для строго одиночного локального прототипа SQLite допустим как временный режим через слой
> доступа к данным, но канон — PostgreSQL.

## 3. Архитектура системы

> Полная единая архитектура (общая с веб‑платформой) описана в
> [docs/architecture/unified-core.md](../architecture/unified-core.md). Ниже — компоненты в
> терминах бота.

Система состоит из трёх независимых компонентов, работающих параллельно:

### 3.1 Компонент 1 — Интерфейс ментора (aiogram)
- Принимает команды только от Telegram ID ментора.
- Ввод и парсинг сигналов.
- Управление учениками.
- Просмотр истории и статистики.

### 3.2 Компонент 2 — Доставка рассылки
- Отправка персональных сообщений ученикам с задержкой 3–5 сек между отправками.
- Логирование результата каждой отправки (sent / failed + причина).
- Идемпотентность: повторный запуск доставки одного сигнала не дублирует сообщения
  (см. [AUDIT.md → A‑06](../audit/AUDIT.md#a-06-идемпотентность-и-ретраи-доставки)).
- **Способ** (userbot Pyrogram или обычный бот) — открытое решение, см. раздел 1.

### 3.3 Компонент 3 — WEEX Sync (APScheduler)
- Каждые N минут (настройка `balance_sync_interval`, дефолт 30) запрашивает аффилиат API WEEX.
- Обновляет баланс, позиции и историю сделок каждого ученика.
- При недоступности данных — уведомляет ментора и ученика.
- Автоматически закрывает сигналы, когда позиция закрыта.

### Поток данных при отправке сигнала

```
Ментор вводит сигнал в aiogram-бот
      ↓
Парсинг сигнала (пара, направление, плечо, вход, стоп, ТП)
      ↓
Запрос цены с WEEX /capi/v3/market/symbolPrice (если вход не указан)
      ↓
Запрос свечей /capi/v3/market/klines (если стоп/ТП не указаны)
      ↓
Авто-расчёт стопа и тейков по режиму
      ↓
Превью ментору → выбор аудитории → подтверждение
      ↓
Для каждого ученика: запрос/чтение баланса → индивидуальный расчёт
      ↓
Доставка ЛС каждому (3-5 сек задержка, идемпотентно)
      ↓
Отчёт ментору: кому отправлено / кому нет и почему
```

## 5. WEEX API интеграция

### 5.1 Базовый домен
- Futures REST API: `https://api-contract.weex.com`
- Affiliate API: `https://api-spot.weex.com`

### 5.2 Аутентификация (HMAC SHA256 + Base64)
Каждый запрос подписывается заголовками:

```
ACCESS-KEY:        Ключ API
ACCESS-SIGN:       base64(hmac_sha256(secret, timestamp+METHOD+path+body))
ACCESS-PASSPHRASE: Пользовательская фраза
ACCESS-TIMESTAMP:  Unix timestamp в миллисекундах
```

### 5.3 Используемые эндпоинты

| Назначение | Метод | Эндпоинт |
|---|---|---|
| Текущая цена пары | GET | `/capi/v3/market/symbolPrice?symbol=BTCUSDT` |
| Свечи для расчёта стопа/ТП | GET | `/capi/v3/market/klines?symbol=X&interval=15m&limit=50` |
| Время сервера WEEX (UTC+8) | GET | `/capi/v3/market/time` |
| Баланс фьючерс. счёта | GET | `/capi/v3/account/balance` (личный ключ) |
| **Баланс реферала по UID** | GET | *закрытый аффилиат‑эндпоинт баланса* (доступ получен) |
| Торговые данные реферала | GET | `/api/v2/rebate/affiliate/getChannelUserTradeAndAsset` |
| Комиссия по рефералам | GET | `/api/v3/rebate/affiliate/getAffiliateCommission` |

### 5.4 Часовой пояс
WEEX работает в **UTC+8**; временные метки от API приходят в UTC+8. **Правило проекта:**
все метки приводятся к **UTC** при сохранении в БД и конвертируются в часовой пояс пользователя
только при отображении. Это исключает рассинхрон между ботом и веб‑платформой
(см. [A‑07](../audit/AUDIT.md#a-07-часовые-пояса)).

### 5.5 Аффилиат API — получение баланса ученика

> ✅ **Статус: доступ получен.** Закрытый эндпоинт баланса рефералов по UID согласован с партнёрским
> менеджером WEEX. Используем его как основной источник `students.balance_usdt`.

Требования к интеграции:
- Реальный закрытый эндпоинт баланса реферала (точный путь и схему ответа — зафиксировать в коде
  и `.env`/конфиге при подключении).
- **Fallback:** если эндпоинт временно недоступен или UID не возвращает баланс — допускается ручной
  ввод/последнее известное значение баланса с пометкой `balance_source = manual` и предупреждением
  ментору. Сигнал на устаревшем балансе помечается в отчёте доставки.
- Все ошибки аффилиат‑API логируются с кодом ответа; UID маскируется в логах.

> 🔧 Формулировка «реализовать заглушку» из v1.0 удалена — доступ есть. Заглушка переквалифицирована
> в полноценный **fallback** ручного баланса на случай сбоев API.

## 6. Режимы торговли и формулы расчёта

> Все формулы — единый источник истины, реализуются в общем пакете `calculator`
> ([unified-core.md](../architecture/unified-core.md#калькулятор)). Денежные расчёты — в `Decimal`
> (Python) / `NUMERIC` (БД).

### 6.1 Умеренный режим
Плечо: до x25. Расчёт маржи по формуле от баланса.

```
margin = balance ** 0.55 * 1.8
```

| Баланс USDT | Маржа $ | % от баланса |
|---|---|---|
| 200 | ~41 | ~20% |
| 500 | ~63 | ~12% |
| 1000 | ~90 | ~9% |
| 2000 | ~128 | ~6.4% |
| 5000 | ~230 | ~4.6% |

Остальные параметры:

```
position_size = margin * leverage
sl_dist%      = moderate_sl_percent (настройка, дефолт 1.5%)
risk_usd      = position_size * sl_dist% / 100
profit_tpN    = position_size * tpN_percent / 100
```

### 6.2 Турбо режим
Плечо: до x400 (ментор задаёт в сигнале, ученик может переопределить своё). Маленькая маржа,
большой объём.

```
margin = min(balance ** 0.6 * 1.2, turbo_margin_cap)   # turbo_margin_cap дефолт 150
```

| Баланс USDT | Маржа $ | Пример x200 — объём |
|---|---|---|
| 200 | ~34 | ~6,800 |
| 500 | ~54 | ~10,800 |
| 1000 | ~76 | ~15,200 |
| 2000 | ~107 | ~21,400 |
| 3000+ | 150 (кап) | ~30,000 |

```
position_size = margin * leverage
sl_dist%      = turbo_sl_percent (настройка, дефолт 0.5%)
risk_usd      = position_size * sl_dist% / 100
profit_tpN    = position_size * tpN_percent / 100
```

> ⚠️ **Предупреждение риска.** Если `risk_usd > 20%` от баланса — бот показывает предупреждение ⚠️
> в сообщении ученику. Для турбо, где риск может равняться всей марже, обязателен явный дисклеймер
> (см. [A‑04 — финансовая ответственность](../audit/AUDIT.md#a-04-финансоваяправовая-ответственность)).

### 6.3 Авто‑расчёт стопа и тейков
Если ментор не указал стоп и/или тейки — бот считает сам:

| Параметр | Умеренный | Турбо |
|---|---|---|
| Стоп‑лосс | −1.5% от входа | −0.5% от входа |
| TP1 | +1.5% | +0.5% |
| TP2 | +3.0% | +1.0% |
| TP3 | +4.5% | +1.5% |

- Направление учитывается: для SHORT стоп **выше** входа, тейки **ниже**.
- Все % настраиваются через `/settings` (таблица `settings`, раздел 4.4).

## 7. Формат ввода сигнала

### 7.1 Правила парсинга
Ментор пишет сигнал в свободной форме. Обязательны только **пара** и **направление** — остальное
опционально. Бот парсит что есть и достраивает остальное.

**Обязательные:** тикер пары (`XLM`, `BTCUSDT`, `ETH`…), направление (`LONG`/`SHORT`/`лонг`/`шорт`).

**Опциональные:**
- Плечо: `20х`, `x20`, `плечо 20` → иначе дефолт из настроек.
- Вход: `0.150`, `вход 0.150` → иначе текущая цена с WEEX.
- Стоп: `стоп 0.145`, `sl 0.145` → иначе бот считает.
- ТП: `тп1 0.153`, `tp1 0.153`, `тейк 0.153` → иначе бот считает.
- Тип маржи: `кросс`/`cross`/`изол`/`isolated` → дефолт `cross`.
- Тип входа: `по рынку`/`market`/`лимит`/`limit` → дефолт `market`.

### 7.2 Примеры форматов

```
# Минимальный (всё авто)
XLM LONG

# С плечом
XLM LONG
Плечо 20х

# С лимитным входом
XLM LONG
Плечо 20х
Вход 0.150
Кросс, лимит

# С ручными тейками
XLM LONG
Плечо 20х
ТП1 0.153
ТП2 0.157
Кросс, по рынку

# Полный вариант
XLM LONG
Плечо 20х
Вход 0.150
Стоп 0.145
ТП1 0.153
ТП2 0.157
ТП3 0.162
Кросс, лимит
```

## 8. Регистрация и онбординг учеников

### 8.1 Поток для ученика

```
/start → "Заявка отправлена ментору"
      ↓
Ментору: уведомление [✅ Принять] [❌ Отклонить]
      ↓ (Принять)
Ученик: "Доступ открыт! Пройдите регистрацию."
      ↓
Шаг 1: язык [🇷🇺 Русский] [🇬🇧 English]
Шаг 2: режим [📊 Умеренный] [⚡ Турбо]
Шаг 3 (только умеренный): риск % (1-5%)
Шаг 4: WEEX UID
      ↓
Проверка UID через аффилиат API
      ↓
Успех: "Регистрация завершена! Баланс: 342 USDT"
Ошибка: "UID не найден, попробуйте снова"
```

При отклонении: «Доступ закрыт. Обратитесь к ментору.»

### 8.2 Настройки ученика (`/settings`)
Сменить режим · сменить риск % (умеренный) · сменить язык · обновить WEEX UID · своё плечо для турбо.

## 10. Команды и интерфейс ментора
Все команды — только для ментора (проверка по Telegram ID).

- **`/signal`** — новый сигнал: парсинг → запрос цены → расчёт стопа/ТП → превью → выбор аудитории
  `[Всем] [Умеренным] [Турбо]` → `[✅ Отправить] [❌ Отменить]` → отчёт о доставке.
- **`/students`** — список учеников; `/activate`, `/deactivate`, `/refresh_balance`, `/student_info`.
- **`/history`** — история сигналов (дата, тикер, направление, кому отправлено, статус).
- **`/settings`** — глобальные настройки (% стопа/ТП, кап маржи, дефолтное плечо, интервал синхр.).
- **`/stats`** — активные ученики, сигналы за период, активные сигналы.

## 12. Уведомления и алерты

**Ментору:** новый `/start`; ошибка доставки; UID не найден; ошибка WEEX API; позиция ученика закрыта.

**Ученику:** доступ одобрен/отклонён; регистрация завершена; UID не найден; ошибка получения баланса.

## 13. Безопасность

- Команды ментора — только от его Telegram ID (в конфиге, не в коде).
- Ученики получают доступ только после подтверждения.
- WEEX API‑ключи — в `.env`, не в БД.
- Сессия userbot (если используется) — в зашифрованном файле.
- Логи без чувствительных данных; UID маскируется.
- Задержка 3–5 сек между отправками.
- Сводные требования безопасности — в [AUDIT.md → A‑08](../audit/AUDIT.md#a-08-безопасность-и-секреты).

## Changelog доработки

Изменения относительно исходного ТЗ v1.0 (обоснования — в [AUDIT.md](../audit/AUDIT.md)):

1. **СУБД: SQLite → PostgreSQL** (A‑03). Единая база с веб‑платформой.
2. **Денежные типы:** `REAL` → `NUMERIC`/`Decimal` для всех сумм (A‑09).
3. **WEEX баланс реферала:** «заглушка» заменена на реальный закрытый эндпоинт (доступ получен),
   заглушка переформулирована как **fallback** ручного баланса; добавлено поле `balance_source` (A‑01).
4. **Способ доставки:** Pyrogram userbot помечен как **открытое решение** с разбором рисков (A‑02).
5. **Идемпотентность доставки:** уникальный индекс `(signal_id, student_id)` + ретраи (A‑06).
6. **Часовые пояса:** единое правило хранения в UTC, отображение в TZ пользователя (A‑07).
7. **Единый источник настроек и шаблонов** для бота и веб‑панели ментора (A‑05).
8. **Калькулятор/WEEX/модели вынесены в общий пакет `core`** (A‑03), бот — клиент ядра.
9. **Усилены дисклеймеры риска** для турбо (A‑04).
10. Структурная чистка: Markdown, кодовые блоки, ссылки на единую архитектуру и аудит.
